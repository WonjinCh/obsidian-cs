# 빌더 패턴

상위개념: [[00_생성 패턴]] | 인덱스: [[디자인패턴 개요]]

---

## 개요

복잡한 객체의 생성 과정과 표현 방법을 완벽히 분리하여, **동일한 생성 절차에서 서로 다른 표현 방식의 복잡한 객체를 만들 수 있게** 하는 유연한 생성 패턴. 수많은 옵셔널 매개변수가 있는 생성자를 우아하게 대체합니다.

---

## 핵심 아이디어

1. `Builder`라는 별도 클래스로 객체 생성 단계를 캡슐화
2. 점층적 생성자 패턴(Telescoping Constructor)과 Java Bean(Setter 체인) 패턴의 장점을 결합
3. 필수(Required) 요소와 선택적(Optional) 요소를 명확히 구분
4. 생성 메서드(`build()`)를 통해 불변 객체(Immutable Object)를 손쉽게 반환

---

## 장단점

| | 특징 |
|---|---|
| 장점 | 복잡한 객체 생성 코드를 숨김으로써(메서드 체이닝) 가독성 극대화, 불변성 보장 |
| 단점 | 수많은 추가적인 Builder 클래스들을 디자인해야 하므로 코드가 방대해짐, 매번 빌더를 거쳐 객체를 생성하는 데 오버헤드가 있음 |

---

## 예시 코드

### Java (Lombok-style)
```java
class Car {
    private String engine;
    private int seats;
    private boolean hasGPS;
    
    // 생성자는 외부 노출 금지
    Car(String engine, int seats, boolean hasGPS) {
        this.engine = engine;
        this.seats = seats;
        this.hasGPS = hasGPS;
    }
}

class CarBuilder {
    private String engine;
    private int seats = 4;       // 기본값
    private boolean hasGPS = false; 

    public CarBuilder engine(String val) { engine = val; return this; }
    public CarBuilder seats(int val)     { seats = val; return this; }
    public CarBuilder gps(boolean val)   { hasGPS = val; return this; }

    public Car build() {
        return new Car(engine, seats, hasGPS);
    }
}

// 클라이언트
Car myCar = new CarBuilder().engine("V8").gps(true).build();
```

---

## 언제 사용하나

- 생성자의 매개변수가 지나치게 많아 클라이언트 코드에서 값을 파악하기 힘들 때.
- 점층적 생성자 패턴을 억제하고 가독성 좋고 안전한(불변) 객체를 만들어야 할 때.
- 특정 과정들을 지연(연기)시키거나 단계적으로 객체를 조립해야 하는 경우 (예: StringBuilder 기반 동적 SQL 생성).

---

## 주의사항

- 아주 단순한 객체의 생성에 빌더를 도입하면 불필요한 클래스 양산으로 시스템이 복잡해집니다.
