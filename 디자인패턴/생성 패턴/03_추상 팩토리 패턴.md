# 추상 팩토리 패턴

상위개념: [[00_생성 패턴]] | 인덱스: [[디자인패턴 개요]]

---

## 개요

구체적인 클래스를 명시하지 않고도 **관련되거나 의존적인 여러 객체의 '군(Family)'을 포괄적으로 묶어 생성하는 추상 인터페이스를 제공**하는 패턴. (즉, 여러 팩토리 메서드를 하나의 그룹으로 묶은 형태)

---

## 핵심 아이디어

1. 제품 집합(Family)과 무관하게 서로 완벽하게 호환됨을 보장
2. 제품군 생성의 모든 단계를 구체화하지 않고 인터페이스만 노출
3. 새로운 종류의 제품을 추가하기엔 번거롭지만(모든 팩토리에 메서드 추가 필요), 제품군 추가는 매우 쉬움

---

## 장단점

| | 특징 |
|---|---|
| 장점 | 각 팩토리 계열끼리의 제품군 간 일관성(상호 호환성) 보장, 단일 책임 원칙 준수 |
| 단점 | 팩토리 인터페이스(부모)가 변경될 경우 수많은 서브클래스에 반영해야 하므로 코드가 복잡하고 무거워짐 |

---

## 예시 코드

### Java
```java
// Abstract Products
interface Button { void paint(); }
interface Checkbox { void paint(); }

// Abstract Factory
interface GUIFactory {
    Button createButton();
    Checkbox createCheckbox();
}

// Concrete Factory
class MacFactory implements GUIFactory {
    public Button createButton() { return new MacButton(); }
    public Checkbox createCheckbox() { return new MacCheckbox(); }
}

// Client
class Application {
    private Button button;
    private Checkbox checkbox;

    public Application(GUIFactory factory) {
        button = factory.createButton();
        checkbox = factory.createCheckbox();
    }
}
```

---

## 언제 사용하나

- 시스템이 여러 제품군(Family) 중 하나를 무조건 선택해서 구성해야 할 때.
- 동종의 객체들이 항상 함께 사용되도록 설계되어 이기종 객체가 섞이는 것을 예방하고 싶을 때 (예: Windows UI 제품군과 Mac UI 제품군을 엮어서 쓰면 에러 발생).

---

## 주의사항

- 제품군 자체는 매우 쉽게 교체 가능(DIP, OCP)하지만, 군 안에 "새로운 유형의 제품(예: ScrollBar)"이 추가되어야 할 경우 인터페이스를 수정해야 하므로 OCP가 일부 위배될 수 있다.
