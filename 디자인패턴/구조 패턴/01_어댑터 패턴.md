# 어댑터 패턴

상위개념: [[00_구조 패턴]] | 인덱스: [[디자인패턴 개요]]

---

## 개요

호환되지 않는 인터페이스를 가진 두 객체가 협업할 수 있도록 변환해 주는 구조 패턴. (실생활의 110V-220V 돼지코나 USB 젠더 역할)

---

## 핵심 아이디어

1. 사용하려는 라이브러리/기존 시스템(Adaptee)의 인터페이스가 현재 시스템이 요구하는 규격(Target)과 다름.
2. 어댑터(Adapter) 클래스를 만들어 Target 인터페이스를 구현하고, 내부에 Adaptee 객체의 레퍼런스를 갖게 함(합성).
3. 클라이언트는 오직 Target 인터페이스로만 소통.

---

## 장단점

| | 특징 |
|---|---|
| 장점 | 기존 코드(Adaptee)를 일절 수정하지 않고 시스템에 통합 가능(OCP 준수), 결합도 저하 |
| 단점 | 새로운 어댑터 클래스, 인터페이스 도입으로 코드 복잡성 증가 |

---

## 예시 코드

### Java
```java
// 기존 타겟 인터페이스
interface Target { void request(); }

// 호환 안 되는 외부 클래스
class Adaptee {
    public void specificRequest() { System.out.println("특수 기능 처리"); }
}

// 변환기
class Adapter implements Target {
    private Adaptee adaptee;
    public Adapter(Adaptee adaptee) { this.adaptee = adaptee; }

    @Override
    public void request() {
        adaptee.specificRequest(); // 호환되지 않는 메서드를 래핑해서 호출
    }
}
```

---

## 언제 사용하나

- 새롭게 도입한 3rd-party 라이브러리를 기존 시스템의 규격에 맞게 꽂아 넣고 싶을 때.
- 인터페이스의 스펙이 달라 구동이 불가능한 레거시 코드를 재사용해야 할 때.

---

## 주의사항

- 레거시 코드를 수정하는 게 낫다면 굳이 어댑터를 만들 필요 없습니다. (어댑터는 '코드 접근이 힘들거나, 손대면 사이드 이펙트가 심할 때' 주로 씁니다)
