# 브릿지 패턴

상위개념: [[00_구조 패턴]] | 인덱스: [[디자인패턴 개요]]

---

## 개요

커다란 클래스나 밀접하게 관련된 클래스들의 집합을 '추상화(Abstraction)' 부분과 '구현(Implementation)' 부분이라는 두 개의 개별 계층으로 나누어 **각각 독립적으로 확장할 수 있도록** 설계하는 패턴.

---

## 핵심 아이디어

1. 기능 계층과 구현 계층을 엄격히 분리 (상속 대신 합성 브릿지로 연결).
2. `모양(원, 사각형) × 색상(빨강, 파랑)`의 경우 4개의 서브클래스 대신, 두 차원을 조합(Bridge)함으로써 파일 개수를 줄일 수 있음.

---

## 장단점

| | 특징 |
|---|---|
| 장점 | 플랫폼 독립적인 추상화 구성, 새로운 추상화나 구현의 추가가 다른 계층에 영향 X (OCP) |
| 단점 | 결합도가 높아야 할 코드에 무리하게 적용 시 코드가 파편화되고 복잡도 상승 |

---

## 예시 코드

### Java
```java
// 구현부 인터페이스 (색상)
interface Color { void fill(); }
class RedColor implements Color { public void fill() { System.out.println("빨간색"); } }

// 추상부 (모양)
abstract class Shape {
    protected Color color; // 구현부 브릿징
    public Shape(Color color) { this.color = color; }
    abstract void draw();
}

// 추상부 구체화
class Circle extends Shape {
    public Circle(Color color) { super(color); }
    public void draw() { System.out.print("원 - "); color.fill(); }
}
```

---

## 언제 사용하나

- 인터페이스(기능)와 그 실제 구현체가 양쪽 다 서브클래스로 자주 확장되어 클래스 폭발이 일어날 가능성이 있을 때.

---

## 주의사항

- 인터페이스와 구현을 초기 설계 때부터 나눌 것을 예상하고 구조를 잡아야 유용합니다.
