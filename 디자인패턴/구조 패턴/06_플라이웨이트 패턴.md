# 플라이웨이트 패턴

상위개념: [[00_구조 패턴]] | 인덱스: [[디자인패턴 개요]]

---

## 개요

매우 많은 수의 유사 객체가 각자 동일하게 가지고 있는 무거운 공통 정보(내재적 상태)를 반복해서 메모리에 할당하는 대신, **하나만 생성(캐싱)하여 여러 객체가 서로 '공유'함으로써 RAM 사용량을 극단적으로 최적화**하는 패턴.

---

## 핵심 아이디어: 상태 분리

1. **내재적 상태 (Intrinsic State):** 객체 내부에 숨겨져 있고, 다른 객체에서 공유될 수 있는 값이 절대 변하지 않는(Immutable) 텍스처 데이터, 이미지 데이터 등.
2. **외재적 상태 (Extrinsic State):** 객체 외부에 있으며 환경적 요인(예: 나무마다 배치된 x, y 좌표값 등)에 따라 변형 가능한 상태. (저장하지 않고 파라미터로 받음)
3. 내재적 상태만 플라이웨이트 객체 내부에 캐싱(맵 컬렉션)하여 공유 사용.

---

## 장단점

| | 특징 |
|---|---|
| 장점 | 시스템의 수백만 개 무거운 객체의 공통 데이터 부하를 해소하고 메모리를 엄청나게 절약 (RAM 초과 방어) |
| 단점 | 코드 설계가 상당히 복잡해지며 외재적 데이터를 파라미터로 계산/전송해야 하므로 CPU 실행 성능에서 지연이 발생할 수 있음 |

---

## 예시 코드

### Java
```java
// Flyweight 모델(내부 불변 상태)
class TreeType { 
    private String name; private String color; // 무겁다고 가정할 내재 속성
    public TreeType(String name, String color) { ... }
    
    // 외부 상태(x, y)는 메서드 파라미터로 주입받아 동작 처리
    public void draw(int x, int y) { ... }
}

// Flyweight Factory (캐싱)
class TreeFactory {
    static Map<String, TreeType> treeTypes = new HashMap<>();
    
    public static TreeType getTreeType(String name, String color) {
        TreeType result = treeTypes.get(name);
        if (result == null) {
            result = new TreeType(name, color);
            treeTypes.put(name, result); // 동일 옵션 재사용 등록
        }
        return result; 
    }
}
```

---

## 언제 사용하나

- 생성해야 하는 객체의 절대적인 양이 너무 많아서, RAM 등 메모리가 심각하게 낭비되거나 OutOfMemory 예외를 일으킬 위기일 때. (예: 워드 프로세서 파일 안의 모든 글자, 게임 내의 모든 입자(파티클)/숲의 나무들 렌더링).

---

## 주의사항

- 플라이웨이트 객체 생성 후 내재된 상태를 변경(Set)하게 되면, 이걸 참조하는 수많은 다른 위치의 클라이언트들의 값도 덩달아 파괴되므로 무조건 Immutable(불변 객체)로 설계해야 합니다.
