# 컴포지트 패턴

상위개념: [[00_구조 패턴]] | 인덱스: [[디자인패턴 개요]]

---

## 개요

객체들을 **트리(Tree) 구조로 구성**하여 부분-전체 계층을 표현하는 패턴. 클라이언트가 트리 내부의 개별 객체(단일체)와 복합 객체(그룹)를 구별 없이 동일한 인터페이스로 다룰 수 있습니다.

---

## 핵심 아이디어

1. 리프(Leaf) 노드와 내부(Composite) 노드의 다형성을 활용(공통 `Component` 인터페이스).
2. 재귀적인 구조 처리로, 부모 노드의 메서드 호출 시 모든 자식 트리 요소에 전달 위임하여 일괄 처리함.

---

## 장단점

| | 특징 |
|---|---|
| 장점 | 복잡한 계층 트리를 구성하는 단일/복합 객체를 통일되게, 쉽게 다룰 수 있음, 새로운 트리를 구성하기 매우 수월(OCP) |
| 단점 | 공통 인터페이스 설계에 모든 일반적인 연산을 끼워 넣으려 하다 보면 단일 객체엔 필요 없는 메서드가 과도하게 생김(의미론적 타입 안정성 위반 요지) |

---

## 예시 코드

### Java
```java
// Component 공통 인터페이스
interface FileSystemNode {
    void printName();
}

// Leaf (말단 단일 객체)
class File implements FileSystemNode {
    private String name;
    public File(String name) { this.name = name; }
    public void printName() { System.out.println("  파일: " + name); }
}

// Composite (복합 그룹 객체)
class Directory implements FileSystemNode {
    private String name;
    private List<FileSystemNode> children = new ArrayList<>();
    
    public Directory(String name) { this.name = name; }
    public void add(FileSystemNode node) { children.add(node); }
    
    public void printName() {
        System.out.println("폴더: " + name);
        // 재귀적 위임
        for (FileSystemNode child : children) {
            child.printName();
        }
    }
}
```

---

## 언제 사용하나

- 대상 도메인이 파일 시스템, UI 계층 구조(DOM 등)처럼 명확한 트리(Tree)형태의 '전체-부분' 계층 체계를 가질 때.
- 클라이언트 코드를 단순화시켜, 대상이 1개인지 배열에 묶인 여러 개인지 신경 쓰지 않게 하고 싶을 때.

---

## 주의사항

- 인터페이스의 책임 범위를 너무 넓게 잡지 말고 런타임에 올바른 노드 요소가 추가되게끔 관리해야 합니다.
