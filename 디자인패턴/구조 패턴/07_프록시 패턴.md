# 프록시 패턴

상위개념: [[00_구조 패턴]] | 인덱스: [[디자인패턴 개요]]

---

## 개요

어떤 다른 무거운 객체(실제 서비스/레코드 대상)에 대한 접근을 은밀하게 가로채고 제어하기 위해, **대상체 도달 전후에 대리자나 자리표시자(Proxy)를 중간 필터 역할로 세우는** 구조 패턴.

---

## 핵심 아이디어

1. 클라이언트는 실제 객체와 프록시 객체를 구분할 수 없음 (동일 인터페이스 구현).
2. 종류가 다양: 가상 프록시(지연 초기화), 보호 프록시(접근 제어), 캐싱 프록시(결과 저장 복원), 원격 프록시(네트워크 통신).
3. 핵심 서비스 객체 실행 직전, 지연 생성/인증/캐시 템플릿과 같은 작업을 수행한 뒤 실제 위임을 넘김.

---

## 장단점

| | 특징 |
|---|---|
| 장점 | 원본 코드(Target)를 절대 건드리지 않고(OCP) 보안, 지연 초기화, 로깅 등 부가 기믹을 삽입할 수 있음 (관심사 분리) |
| 단점 | 여러 프록시 클래스가 체이닝되면 시스템 응답 속도가 느려지며, 코드 이해의 직관성이 저하|

---

## 예시 코드

### Java
```java
// 대상과 프록시가 구현할 공통 인터페이스
interface VideoDownloader { void downloadVideo(String videoName); }

// RealSubject (실제 다운로더)
class RealVideoDownloader implements VideoDownloader {
    public void downloadVideo(String v) { System.out.println(v + " 네트워크로 실제 다운로드..."); }
}

// Proxy
class ProxyVideoDownloader implements VideoDownloader {
    private RealVideoDownloader real;

    @Override
    public void downloadVideo(String v) {
        // 객체가 필요하기 전까진 안 만들음(가상 프록시 - Lazy)
        if (real == null) real = new RealVideoDownloader();
        
        System.out.println("대리인: 사용자 보안 권한 검사 통과");
        
        real.downloadVideo(v); // 진짜 객체에 로직 위임
        
        System.out.println("대리인: 로깅 기록 갱신 성공");
    }
}
```

---

## 언제 사용하나

- 거대한 초기화 메모리 비용을 방지하고 싶을 때 (Lazy Initialization).
- Spring AOP 등 인프라스트럭처 레벨의 횡단 관심사(트랜잭션 바인딩, 인증, 로깅, 캐시 - @Cacheable 등) 처리를 비즈니스 코드와 분리하고 싶을 때.

---

## 주의사항

- 수없이 잦은 객체 생성이 이뤄지는 곳에서 프록시 래핑을 남발하면 객체 생성 비용 프레임 오버헤드가 누적됩니다.
