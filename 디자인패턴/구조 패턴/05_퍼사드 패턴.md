# 퍼사드 패턴

상위개념: [[00_구조 패턴]] | 인덱스: [[디자인패턴 개요]]

---

## 개요

복잡하게 얽힌 서브시스템(라이브러리 세트, 프레임워크 클래스들)의 덩어리에 대해, 클라이언트들이 쉽게 사용할 수 있는 **단순화된 상위 수준의 단일 통합 인터페이스 창구(정면, Facade)**를 제공하는 패턴.

---

## 핵심 아이디어

1. 복잡성을 외부로 드러내지 않기 위한 껍데기를 세움 (컴퓨터 조립 부품들의 실행 순서를 숨긴 전원 버튼 한 개 구축).
2. 시스템 파사드는 서브시스템들을 직접 참조하여 순서대로 동작들을 위임, 캡슐화.

---

## 장단점

| | 특징 |
|---|---|
| 장점 | 클라이언트가 서브시스템의 복잡한 클래스와 직접 결합되는 것을 완벽히 방어(의존성 저하), 호출 단순화 제공 |
| 단점 | 파사드 클래스 1개 자체가 애플리케이션의 모든 주요 요소들에 결합되어 있는 거대한 "God Object(전지전능)"가 될 위험 |

---

## 예시 코드

### Java
```java
// 서브시스템들
class CPU { public void freeze() { System.out.println("CPU 멈춤"); } }
class Memory { public void load() { System.out.println("메모리 로딩"); } }
class HardDrive { public void read() { System.out.println("디스크 읽기"); } }

// 파사드 (통합 인터페이스)
class ComputerFacade {
    private CPU cpu; private Memory memory; private HardDrive hardDrive;

    public ComputerFacade() {
        this.cpu = new CPU(); this.memory = new Memory(); this.hardDrive = new HardDrive();
    }

    // 클라이언트는 내부 복잡도를 무시하고 이 버튼 한 개만 노출받음
    public void start() {
        cpu.freeze();
        hardDrive.read();
        memory.load();
    }
}
```

---

## 언제 사용하나

- 3rd-party 외부 라이브러리나 규모가 큰 컴포넌트 내부 사용법이 너무 무겁고 복잡해서 사용하기 난해할 때.
- 계층화(Layered) 아키텍처에서 각 계층의 결합도를 낮추기 위한 진입점 설정이 필요할 때.

---

## 주의사항

- 서브시스템 내의 세부적인 기능 제어가 필요할 때는 클라이언트가 파사드를 우회하여 서브 클래스에 직접 접근해야만 할 수도 있습니다. 단지 편리한 진입점이지 봉쇄막이 아닙니다.
