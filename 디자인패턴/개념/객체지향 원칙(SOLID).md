# 객체지향 원칙 (SOLID)

상위개념: [[디자인패턴 개요]] | 인덱스: [[디자인패턴 개요]]

---

## 개요

로버트 C. 마틴(Robert C. Martin, Uncle Bob)이 정리한, 소프트웨어를 **유연하고 유지 보수하기 쉽게** 만들기 위한 5가지 객체지향 설계 원칙.

---

## 원칙 목록

### SRP — Single Responsibility Principle (단일 책임 원칙)

> 하나의 클래스는 단 하나의 책임(이유)만 가져야 한다.

- 클래스가 변경되어야 하는 이유는 오직 하나여야 함
- 책임을 분리하면 변경 파급 효과가 줄어 유지보수 용이

### OCP — Open-Closed Principle (개방-폐쇄 원칙)

> 확장에는 열려 있고, 수정에는 닫혀 있어야 한다.

- 기존 코드를 변경하지 않고 기능 추가가 가능해야 함
- 인터페이스와 추상화로 다형성 활용 → [[09_전략 패턴]] 이 대표 예시

### LSP — Liskov Substitution Principle (리스코프 치환 원칙)

> 자식 클래스는 언제나 부모 클래스를 대체할 수 있어야 한다.

- 서브타입은 기반 타입의 규약(행동, 전제조건, 사후조건)을 어겨서는 안 됨
- 자식 클래스의 오버라이딩이 부모의 의도를 벗어나면 위반

### ISP — Interface Segregation Principle (인터페이스 분리 원칙)

> 클라이언트는 자신이 사용하지 않는 인터페이스에 의존하면 안 된다.

- 거대한 범용 인터페이스보다 특화된 여러 개의 작은 인터페이스로 분리
- "뚱뚱한 인터페이스"를 피하는 것이 핵심

### DIP — Dependency Inversion Principle (의존성 역전 원칙)

> 고수준 모듈은 저수준 모듈에 의존하면 안 된다. 둘 다 추상화에 의존해야 한다.

- 구체 클래스(Low Level) 가 아닌 인터페이스/추상 클래스(Abstraction)에 의존
- 의존성 주입(DI, Dependency Injection)으로 실현

---

## 핵심 암기 표

| 원칙 | 키워드 | 위반 신호 |
|------|--------|----------|
| SRP | 책임 1개 | 클래스가 너무 뚱뚱함 |
| OCP | 확장 = 추가, 수정 금지 | if 분기로 매번 내부 수정 |
| LSP | 자식 = 부모 대체 가능 | 자식 오버라이딩으로 행동 역전 |
| ISP | 필요한 인터페이스만 | 쓰지도 않는 메서드를 강제 구현 |
| DIP | 추상화에 의존 | `new 구체클래스()` 하드코딩 남발 |
