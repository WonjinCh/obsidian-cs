# 비지터 패턴

상위개념: [[00_행위 패턴]] | 인덱스: [[디자인패턴 개요]]

---

## 개요

기존 객체 구조(Element 클래스)의 코드를 수정하지 않고, **새로운 연산(방문 로직)을 별도의 방문자(Visitor) 객체 안에 추가**하는 패턴. 데이터와 연산을 완전히 분리합니다.

---

## 핵심 아이디어

1. 각 Element는 `accept(Visitor v)` 하나만 노출
2. Visitor는 `visit(ElementA a)`, `visit(ElementB b)` 처럼 각 Element 타입별 로직을 구현
3. 새로운 연산이 필요하면 새 Visitor 클래스만 추가하면 됨 (Element 코드 불변)

---

## 예시 코드

### Java
```java
// 방문 대상 인터페이스
interface Element { void accept(Visitor v); }

class Circle implements Element {
    double radius;
    public Circle(double r) { radius = r; }
    public void accept(Visitor v) { v.visitCircle(this); }
}

// 방문자 인터페이스
interface Visitor {
    void visitCircle(Circle c);
}

// 구체 방문자 (새 연산 추가)
class AreaCalculator implements Visitor {
    public void visitCircle(Circle c) {
        System.out.println("원 넓이: " + Math.PI * c.radius * c.radius);
    }
}
```

---

## 언제 사용하나

- 구조(클래스)는 변하지 않는데 그 위에서 수행해야 할 연산들이 계속 추가되는 경우
- AST(추상 구문 트리) 순회 → 타입 체크, 코드 최적화, 렌더링 등을 각각 다른 Visitor로 처리

---

## 주의사항

- Element 종류가 새로 추가되면 모든 Visitor에 해당 `visit()` 메서드를 추가해야 하므로, Element 구조가 자주 바뀌는 시스템에는 부적합
