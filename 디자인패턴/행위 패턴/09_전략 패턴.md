# 전략 패턴

상위개념: [[00_행위 패턴]] | 인덱스: [[디자인패턴 개요]]

---

## 개요

동일 계열의 **알고리즘(전략)을 캡슐화하여 언제든 교체 가능**하게 만드는 패턴. 클라이언트가 구체적인 알고리즘을 모른 채 인터페이스에만 의존해 사용합니다.

---

## 핵심 아이디어

1. 교체될 수 있는 알고리즘을 `Strategy` 인터페이스로 추상화
2. Context 클래스는 Strategy를 합성으로 참조하고 실행을 위임
3. 런타임에 `setStrategy()`로 알고리즘을 자유롭게 교체

---

## 자료구조

- [[객체지향 원칙(SOLID)]] — OCP 원칙의 교과서적 구현

---

## 예시 코드

### Java
```java
interface SortStrategy { void sort(int[] arr); }

class BubbleSort implements SortStrategy {
    public void sort(int[] arr) { /* 버블 정렬 구현 */ }
}
class QuickSort implements SortStrategy {
    public void sort(int[] arr) { /* 퀵 정렬 구현 */ }
}

class Sorter {
    private SortStrategy strategy;
    public void setStrategy(SortStrategy s) { strategy = s; }
    public void sort(int[] arr) { strategy.sort(arr); }
}

// 클라이언트
Sorter sorter = new Sorter();
sorter.setStrategy(new QuickSort());
sorter.sort(data);
```

---

## 언제 사용하나

- 결제 수단(카드/페이/쿠폰), 할인 정책, 정렬 알고리즘처럼 동일 역할을 하는 여러 구현체 중 하나를 선택해야 할 때
- 조건문으로 알고리즘을 분기하는 코드(`if 카드결제... else 페이결제...`)를 제거하고 싶을 때

---

## 주의사항

- 알고리즘 종류가 2~3개 정도로 매우 적으면 람다/익명 클래스로 간결하게 처리하는 게 나을 수 있음
